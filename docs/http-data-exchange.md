# HTTP и обмен данными

Этот документ объясняет HTTP без привязки к конкретной библиотеке Python.

Стартовая идея:

```text
HTTP - это протокол обмена данными между клиентом и сервером.
```

Протокол - это набор правил. Он нужен, чтобы разные программы понимали друг друга.

## 1. Главные роли

В HTTP почти всегда есть две роли:

| Роль | Кто это | Что делает |
| --- | --- | --- |
| Клиент | Браузер, мобильное приложение, JavaScript на странице | Начинает обмен и отправляет запрос |
| Сервер | Программа на удаленном компьютере | Принимает запрос, выполняет работу и возвращает ответ |

Схема:

```text
┌──────────────┐        HTTP-запрос         ┌──────────────┐
│   Клиент     │  ----------------------->  │    Сервер    │
│   Браузер    │                            │ Python/Node/ │
│ или приложение│  <----------------------- │   другой код │
└──────────────┘        HTTP-ответ          └──────────────┘
```

Важно: обмен обычно начинает клиент.

```text
клиент спросил -> сервер ответил
клиент снова спросил -> сервер снова ответил
```

## 2. Зачем нужен HTTP

HTTP нужен не только для "открыть сайт".

Через HTTP можно:

- загрузить страницу;
- получить список товаров;
- отправить ответ в игре;
- сохранить заметку;
- узнать прогноз погоды;
- отправить форму регистрации;
- получить новые сообщения в чате;
- сохранить общий счет игрока.

Во всех случаях идея одна:

```text
клиент отправляет запрос с данными
сервер возвращает ответ с данными
```

## 3. Что такое запрос

HTTP-запрос - это сообщение от клиента к серверу.

В запросе обычно есть:

| Часть | Пример | Смысл |
| --- | --- | --- |
| Метод | `GET`, `POST` | Какое действие хочет клиент |
| Путь | `/api/start` | Куда именно обращаются на сервере |
| Заголовки | `Content-Type: application/json` | Служебная информация |
| Тело | `{"name": "Аня"}` | Данные, которые клиент отправляет серверу |

Схема запроса:

```text
┌────────────────────────────────────────────┐
│ HTTP-запрос                                │
├────────────────────────────────────────────┤
│ Метод: GET или POST                        │
│ Путь: /api/start                           │
│ Заголовки: Content-Type, Cookie, ...       │
│ Тело: JSON, текст, файл или пусто          │
└────────────────────────────────────────────┘
```

## 4. Что такое ответ

HTTP-ответ - это сообщение от сервера к клиенту.

В ответе обычно есть:

| Часть | Пример | Смысл |
| --- | --- | --- |
| Статус | `200 OK` | Что получилось |
| Заголовки | `Content-Type: application/json` | Что лежит в ответе |
| Тело | `{"score": 5}` | Данные, которые сервер вернул клиенту |

Схема ответа:

```text
┌────────────────────────────────────────────┐
│ HTTP-ответ                                 │
├────────────────────────────────────────────┤
│ Статус: 200 OK или 404 Not Found           │
│ Заголовки: Content-Type, Content-Length    │
│ Тело: HTML, JSON, картинка, текст          │
└────────────────────────────────────────────┘
```

## 5. Пример 1: открыть страницу сайта

Пользователь вводит адрес:

```text
https://example.com/
```

Браузер отправляет запрос:

```text
GET /
```

Сервер возвращает HTML:

```text
200 OK
Content-Type: text/html

<!doctype html>
<html>...</html>
```

Схема:

```text
Пользователь
    │
    │ вводит адрес
    v
┌──────────────┐       GET /        ┌──────────────┐
│   Браузер    │ -----------------> │  Веб-сервер  │
└──────────────┘                    └──────────────┘
       ^                                   │
       │        HTML-страница              │
       └-----------------------------------┘
```

После HTML браузер часто отправляет новые запросы:

```text
GET /style.css
GET /game.js
GET /logo.png
```

То есть одна открытая страница может вызвать несколько HTTP-обменов.

## 6. Пример 2: узнать погоду

Допустим, мобильное приложение погоды хочет узнать температуру в Москве.

Запрос:

```text
GET /api/weather?city=Moscow
```

Ответ:

```json
{
  "city": "Moscow",
  "temperature": -3,
  "condition": "snow"
}
```

Схема:

```text
┌─────────────────────┐
│ Приложение погоды   │
└──────────┬──────────┘
           │ GET /api/weather?city=Moscow
           v
┌─────────────────────┐
│ Сервер погоды       │
└──────────┬──────────┘
           │ JSON с температурой
           v
┌─────────────────────┐
│ Экран с прогнозом   │
└─────────────────────┘
```

Здесь клиент ничего не меняет на сервере. Он только спрашивает данные.

Для таких запросов обычно используют `GET`.

## 7. Пример 3: отправить ответ в игре

В нашей игре браузер показывает пример:

```text
7 * 8
```

Игрок вводит:

```text
56
```

JavaScript отправляет серверу запрос:

```text
POST /api/answer
Content-Type: application/json

{
  "game_id": "abc123",
  "problem_id": "p1",
  "answer": "56"
}
```

Сервер проверяет ответ и возвращает JSON:

```json
{
  "ok": true,
  "correct": true,
  "score": 1,
  "problem": {
    "id": "p2",
    "text": "9 + 4"
  }
}
```

Схема:

```text
┌──────────────┐
│   Браузер    │
│ Игрок: 56    │
└──────┬───────┘
       │ POST /api/answer
       │ JSON: game_id, problem_id, answer
       v
┌──────────────┐
│ Python-сервер│
│ проверяет    │
│ ответ        │
└──────┬───────┘
       │ JSON: correct, score, new problem
       v
┌──────────────┐
│   Браузер    │
│ новый пример │
└──────────────┘
```

Здесь клиент отправляет данные, которые меняют состояние игры.

Для таких запросов обычно используют `POST`.

## 8. Пример 4: сохранить заметку

Пользователь написал заметку:

```text
Купить хлеб
```

Клиент отправляет:

```text
POST /api/notes
Content-Type: application/json

{
  "text": "Купить хлеб"
}
```

Сервер сохраняет заметку и отвечает:

```json
{
  "ok": true,
  "id": 17
}
```

Схема:

```text
┌──────────────┐     POST /api/notes      ┌──────────────┐
│   Клиент     │ -----------------------> │    Сервер    │
│ текст заметки│                          │ сохраняет    │
└──────────────┘ <----------------------- └──────────────┘
                    ok + id заметки
```

Это полезный реальный обмен: клиент просит сервер сохранить данные.

## 9. Пример 5: оформить заказ

Интернет-магазин отправляет заказ на сервер:

```text
POST /api/orders
Content-Type: application/json

{
  "items": [
    {"name": "Клавиатура", "count": 1},
    {"name": "Мышь", "count": 1}
  ],
  "address": "Москва, улица Примерная, 1"
}
```

Сервер проверяет товары, создает заказ и отвечает:

```json
{
  "ok": true,
  "order_id": 105,
  "status": "created"
}
```

Схема:

```text
┌──────────────┐
│ Сайт магазина│
└──────┬───────┘
       │ POST /api/orders
       │ товары + адрес
       v
┌──────────────┐
│ Сервер       │
│ магазина     │
└──────┬───────┘
       │ order_id + status
       v
┌──────────────┐
│ Сайт показал │
│ номер заказа │
└──────────────┘
```

## 10. GET и POST простыми словами

Для первого урока достаточно двух методов.

| Метод | Простое объяснение | Пример |
| --- | --- | --- |
| `GET` | Получить данные | открыть страницу, получить счет игроков |
| `POST` | Отправить данные | начать игру, отправить ответ, сохранить заметку |

Правило для новичка:

```text
GET - хочу получить
POST - хочу отправить
```

Это не все методы HTTP, но для первой веб-игры этого достаточно.

## 11. Что такое JSON

JSON - это текстовый формат для обмена данными.

Он похож на словари и списки Python.

Python:

```python
player = {
    "name": "Аня",
    "score": 5
}
```

JSON:

```json
{
  "name": "Аня",
  "score": 5
}
```

Почему JSON удобен:

- его легко читать человеку;
- его понимает браузер;
- его понимает Python;
- его используют почти все веб-приложения.

Схема обмена JSON:

```text
Объект JavaScript
       │
       │ JSON.stringify(...)
       v
JSON-текст по HTTP
       │
       │ json.loads(...)
       v
Словарь Python
```

В обратную сторону:

```text
Словарь Python
       │
       │ json.dumps(...)
       v
JSON-текст по HTTP
       │
       │ response.json()
       v
Объект JavaScript
```

## 12. Что такое статус ответа

Статус - это короткий результат запроса.

| Статус | Что значит | Пример |
| --- | --- | --- |
| `200 OK` | Все получилось | Сервер отдал страницу |
| `400 Bad Request` | Клиент отправил плохие данные | Вместо числа пришел текст |
| `404 Not Found` | Такого URL нет | `/unknown-page` |
| `500 Internal Server Error` | Ошибка внутри сервера | В коде сервера случилась ошибка |

Для ученика важно:

```text
статус говорит, как закончился запрос
тело ответа содержит подробности
```

## 13. Что такое заголовки

Заголовки - это служебная информация.

Пример заголовка запроса:

```text
Content-Type: application/json
```

Он говорит серверу:

```text
я отправляю JSON
```

Пример заголовка ответа:

```text
Content-Type: text/html; charset=utf-8
```

Он говорит браузеру:

```text
сервер вернул HTML
текст закодирован в UTF-8
```

Схема:

```text
┌──────────────────────────────┐
│ Заголовки                    │
├──────────────────────────────┤
│ объясняют, как понимать тело │
└──────────────────────────────┘
```

## 14. Что такое тело запроса и ответа

Тело - это основное содержимое сообщения.

В запросе тело может содержать:

- имя игрока;
- ответ на пример;
- текст заметки;
- список товаров;
- файл.

В ответе тело может содержать:

- HTML-страницу;
- JSON с результатом;
- картинку;
- текст ошибки;
- CSS или JavaScript.

Схема:

```text
HTTP-сообщение
├─ служебная часть: метод, путь, статус, заголовки
└─ тело: основные данные
```

## 15. Почему серверу нужен URL

URL помогает серверу понять, какую работу нужно выполнить.

В нашей игре:

| URL | Что значит |
| --- | --- |
| `/` | Отдать страницу игры |
| `/api/start` | Начать новую игру |
| `/api/answer` | Проверить ответ |
| `/api/finish` | Завершить игру |
| `/api/leaderboard` | Вернуть общий счет |

Схема:

```text
URL                 Python-код
-----------------------------------------
/                   отдать index.html
/api/start          создать игру
/api/answer         проверить ответ
/api/finish         сохранить счет
/api/leaderboard    прочитать счет из файла
```

## 16. Один обмен и много обменов

Один HTTP-обмен:

```text
запрос -> ответ
```

Но приложение обычно делает много таких обменов.

Открытие игры:

```text
GET /          -> HTML
GET /style.css -> CSS
GET /game.js   -> JavaScript
```

Игра:

```text
POST /api/start  -> первая задача
POST /api/answer -> результат и новая задача
POST /api/answer -> результат и новая задача
POST /api/finish -> сохранение счета
```

Большое приложение - это много маленьких HTTP-обменов.

## 17. HTTP не помнит прошлые запросы сам

HTTP-запрос сам по себе отдельный.

Сервер получил запрос, ответил, и этот обмен закончился.

Поэтому приложения сами добавляют способы помнить контекст.

В нашей игре для этого есть:

```text
game_id
problem_id
```

Схема:

```text
POST /api/start
сервер создал game_id = abc123

POST /api/answer
браузер отправил game_id = abc123
сервер понял, к какой игре относится ответ
```

Без `game_id` сервер не понял бы, какой игрок отвечает на пример.

## 18. Общая схема веб-приложения

```text
┌──────────────┐
│ Пользователь │
└──────┬───────┘
       │ нажимает кнопки, вводит текст
       v
┌──────────────┐
│ Браузер      │
│ HTML/CSS/JS  │
└──────┬───────┘
       │ HTTP-запросы
       v
┌──────────────┐
│ Сервер       │
│ Python-код   │
└──────┬───────┘
       │ читает/пишет данные
       v
┌──────────────┐
│ Файл или БД  │
└──────────────┘
```

В нашем учебном проекте вместо базы данных используется текстовый файл:

```text
data/scores.txt
```

## 19. Что важно запомнить

Главная формула:

```text
клиент -> HTTP-запрос -> сервер
клиент <- HTTP-ответ  <- сервер
```

Запрос отвечает на вопросы:

```text
какой метод?
какой URL?
какие данные отправили?
```

Ответ отвечает на вопросы:

```text
получилось или нет?
какой тип данных вернулся?
какие данные вернулись?
```

Для этой игры:

```text
браузер просит страницу
браузер отправляет имя
браузер отправляет ответы
сервер проверяет ответы
сервер сохраняет счет
браузер показывает результат
```

Это и есть полезный обмен данными через HTTP.

После этого можно перейти к `docs/js-client-server-examples.md`: там показано, как именно JavaScript из этого проекта отправляет HTTP-запросы через `fetch`, а `server.py` обрабатывает эти запросы.
